Your workflow: Session 1 = Systems Map (once). Session 2 per subsystem = Audit → Planning (same session). Session 3 per subsystem = Implementation (new session). Session 4 = Regression Check + Reflection (new session). Session 5 = Verification Pass (fresh session, no implementation context — produces VERIFICATION BLOCK). Key habit: Paste the Systems Map summary at the top of every follow-on session. Paste any Follow-On Audit Items at the top of the next audit for the flagged module.
Cycle Rotation
Cycle 1: Subsystem A→ Cycle 2: Subsystem B→ Cycle 3: Subsystem C→ Cycle 4: Seams & Invariants→ Cycle 5: Subsystem D→ …
Each subsystem cycle = audit + implement + verify. The Seams & Invariants cycle runs every 3–4 subsystem cycles, has no implementation phase, and produces an updated invariant library. It audits the seams between subsystems — the things no vertical audit can own.
§0
Systems Map
Purpose: Orient Claude Code to the codebase before any deep audit. Entry points and relationships only — no deep implementation dives. Run once at project start or after a major restructure. Do not audit in this session.
Systems Map Prompt
Session 0 only 
Do not run audits or recommendations in this session.
Do not make any changes to any files during this session.

Read the following files carefully in this order:
1. CLAUDE.md
2. README (and any supplementary docs in /docs if present)
3. Roadmap file
4. All top-level index, entry-point, and configuration files

Then build a systems map of this codebase. Include:

- Major systems/modules and their responsibilities (one sentence each)
- How data flows between them (inputs → processing → outputs for each major path)
- External dependencies and integrations (APIs, databases, services)
- Authentication and security surface (where auth is enforced, where data is sensitive)
- Anything in CLAUDE.md or documentation that appears to conflict with or be absent from the actual codebase
- Inter-module dependency map: for each major module, what does it export (functions, classes, constants, types), and which other modules consume those exports. Format as a simple list: [Module] → exports [X] → consumed by [Y, Z]

Do NOT read deeply into implementation files yet. Stay at the architectural level.

End with:
- A ranked list of the 3-5 highest-complexity subsystems (most likely to contain hidden issues)
- A ranked list of the 3-5 highest-risk subsystems (most likely to cause problems if broken)
- Any immediate red flags visible from entry points and configuration alone

Note your confidence level on each part of the map (high / medium / low) based on how much of the relevant code you were actually able to read.
§1
Layered Audit
Purpose: Deep audit of one scoped subsystem group per session. Reference the systems map from §0 each time. Produces a Session Handoff Block used as input for §2 Planning.
📌
Paste the Systems Map summary at the top of each audit session before this prompt. 3–5 bullets is enough. One subsystem group per session — scope creep produces shallow findings.
Subsystem Groups Click a row to pre-fill the scope field
Layered Audit Prompt
Session 1+ 
Here is the systems map from a prior session: [PASTE SYSTEMS MAP SUMMARY HERE]
Do not make any changes to any files during this session.

This session's audit scope: [SUBSYSTEM GROUP NAME]
Relevant files/directories: [LIST RELEVANT FILES AND DIRECTORIES]

[OPTIONAL: PASTE ANY FOLLOW-ON AUDIT ITEMS FROM A PRIOR REGRESSION CHECK THAT APPLY TO THIS SCOPE — check these first before running the full audit]

Audit this scope thoroughly. Flag:
- Bugs and logic errors
- Dead or unused code (functions, variables, imports never called or referenced)
- Stale TODOs, commented-out code, and placeholder logic left in production paths
- Hardcoded values that should be config or environment variables
- Security concerns specific to this module (auth gaps, unvalidated inputs, exposed sensitive data)
- Inconsistencies between documentation/CLAUDE.md and actual implementation
- Code quality issues: overly complex functions, poor separation of concerns, naming that obscures intent
- Anything that will compound into a larger problem if not addressed before this module scales

For each finding:
- State the issue clearly in one or two sentences
- Cite the file and approximate location (function name or line range)
- Rate severity: Critical / High / Medium / Low
- State your confidence level: High / Medium / Low (flag if you're inferring from limited context)
- Add a rough effort signal for the fix: S (< 2 hours) / M (half-day to 2 days) / L (3+ days)

End with:
- Top findings by impact — the 5 findings most likely to cause active breakage, data loss, or a security/compliance failure. Note: severity label and impact rank can differ; a High-severity compliance bug may outrank a Critical-severity code quality issue.
- Top 5 highest-leverage improvements (things that would most improve velocity or reliability if addressed)
- Any dependencies or interactions with OTHER subsystems that this audit surfaced

Then produce a SESSION HANDOFF BLOCK formatted exactly as shown below.
This block will be pasted directly into the next planning session — keep it structured, compressed, and complete. Do not add prose or commentary outside the block format.

[ID] | [File: function/line] | [Severity] | [Confidence] | [Effort: S/M/L] | [One-line description]
(repeat for each finding)

High-leverage: [finding IDs]

§1s
Seams & Invariants Audit
Purpose: Audit the seams between subsystems and the invariants that must hold across them — the things no vertical subsystem audit can own. Runs every 3–4 subsystem cycles. No implementation phase. Produces an updated invariant library instead of code changes.
⚠
This is an audit-only cycle. No code changes are made. The output is an updated invariant library and a set of cross-cutting observations that feed into the next subsystem audit's scope and the next Health Synthesis. Do not combine this with a subsystem implementation cycle.

§2

      
Planning & Recommendations

      
Purpose: Translate audit findings into a prioritized action plan. Run at the end of the same audit session or in a fresh session. Produces an Implementation Handoff Block.

    

    
📌
Same session (recommended): Paste this prompt immediately after the audit handoff block is produced — Claude Code uses the handoff block already in context, no re-pasting needed. Fresh session: Paste the Session Handoff Block before this prompt. Start fresh only if the audit session exhausted context (typically 100k+ line codebases with deep subsystems).

    

      

        
Planning Prompt

        

          End of audit session or fresh session
          
          
        

      

      

      
Do not make any changes to any files during this session.

--- SESSION CONTEXT ---
If continuing in the same session as the audit: the handoff block above is your input — proceed directly.
If starting a fresh session: paste the Session Handoff Block from the audit here before continuing.
[PASTE SESSION HANDOFF BLOCK HERE IF FRESH SESSION — otherwise delete this line]
--- END SESSION CONTEXT ---

Based on these findings, produce a prioritized action plan for the [SUBSYSTEM GROUP] scope.

For each recommended action:
- Assign an Action ID (A1, A2, A3...)
- State what to do (specific and concrete — not "improve error handling" but "add try/catch to the transcription callback in [file] and route errors to the audit log")
- Link it back to the finding ID(s) it addresses
- Estimate effort: Small (< 2 hours) / Medium (half-day to 2 days) / Large (3+ days)
- Classify cross-module risk: Low / High / Very High
- Note any prerequisites: actions that must complete first, or other modules that need updating alongside this change

Organize the plan into:
1. Do immediately — critical issues, quick wins, or anything that blocks other work
2. Do this week — high-value improvements that are well-scoped
3. Defer but schedule — important but not urgent; needs planning or has dependencies

If this audit produced more than ~15 findings, split the implementation actions into two explicit batches: Batch 1 (P0/Critical + highest-compliance-risk items) and Batch 2 (everything else). Note the batch split in the handoff block.

Then provide:
- Any findings that should be escalated to the roadmap (too large or structural for a sprint)
- Architectural decisions needed before implementation — list each separately with: the specific question to resolve, the options and their tradeoffs, a recommendation, and whether implementation should be blocked until the decision is made or if a safe default can proceed now
- Any actions that would be good candidates for automation or scripting rather than manual fixes

Then produce a DOCUMENTATION UPDATE CHECKLIST listing every doc that will need updating as a result of the planned actions. For each entry note which action(s) trigger the update and what specifically needs to change. Use this format:

DOCUMENTATION UPDATE CHECKLIST:
[ ] CLAUDE.md — [what needs updating and why]
[ ] README — [what needs updating and why]
[ ] Roadmap — [what needs updating and why]
[ ] Inline code comments — [which files/functions]
[ ] Other: [filename] — [reason]
(omit any file that requires no changes)

Then produce an IMPLEMENTATION HANDOFF BLOCK formatted exactly as shown below.
This block will be pasted into the implementation session — keep it structured and complete.

---IMPLEMENTATION HANDOFF BLOCK---
Scope: [subsystem group name]
Systems map reference: [confirm whether systems map was available in this session]
Batch: [1 of 1 / 1 of 2 / 2 of 2 — omit if no batch split]

ACTIONS TO IMPLEMENT:
[ID] | [File: function/area] | [Effort] | [Cross-module risk] | [One-line description] | [Prerequisite action IDs if any]
(repeat for each action)

HIGH/VERY HIGH RISK ACTIONS (require dependency check before implementation):
[List action IDs and the specific exports/interfaces they touch]

IMPLEMENT IN THIS ORDER: [ordered list of action IDs accounting for prerequisites]
ORDERING RATIONALE: [1-2 sentences — why this order, which items must land together in the same commit/PR, and whether high-risk items depend on low-risk items being committed first]
---END IMPLEMENTATION HANDOFF BLOCK---

    

  


  
  

    
§3

      
Implementation Session

      
Purpose: Dedicated session for writing, editing, and deleting code. Always a separate session from audit/planning — starts with the systems map and Implementation Handoff Block. Produces an Implementation Summary Block.

    

    
📌
Open a new session for implementation. Paste the systems map summary first, then the Implementation Handoff Block from §2. For High/Very High risk actions, run §4 Pre-Implementation Check at the start of this session before touching those specific actions.

    

      

        
Implementation Session Prompt

        

          New session — code-writing
          
          
        

      

      

      
[PASTE SYSTEMS MAP SUMMARY HERE — same 3-5 bullet format used in the audit session]

[PASTE IMPLEMENTATION HANDOFF BLOCK FROM PLANNING SESSION HERE]

Before starting: review the HIGH/VERY HIGH RISK ACTIONS listed in the handoff block above. For any of those actions, run the Pre-Implementation Dependency Check (§4) now and paste the results here before implementing those specific actions. Low-risk actions may proceed without a dep check.

You are now in an implementation session. The actions above are the agreed scope for this session.

Rules for this session:
- Implement only the actions listed in the handoff block. Do not fix, refactor, or improve anything outside this scope, even if you notice other issues. Flag those separately at the end.
- Work through actions in the order specified in the handoff block unless a blocker requires reordering — if so, say why before reordering.
- Before implementing any action rated High or Very High cross-module risk, confirm your understanding of the change and its intended effect. Wait for acknowledgement before proceeding.
- If a finding turns out to be more complex than the effort estimate suggested, stop and describe what you found before continuing. Do not improvise a larger solution without discussion.
- If implementing an action would require touching files outside the listed scope, stop and flag it rather than proceeding.
- After completing each action, note briefly: what was changed, what file(s) were touched, and whether anything unexpected was encountered.

When all actions are complete, produce an IMPLEMENTATION SUMMARY BLOCK:

---IMPLEMENTATION SUMMARY BLOCK---
Session scope: [subsystem group]
Actions completed: [list action IDs]
Actions not completed (if any): [list with reason]

CHANGES MADE:
[Action ID] | [File(s) modified] | [Brief description of what changed]
(repeat for each completed action)

UNEXPECTED FINDINGS DURING IMPLEMENTATION:
- [anything discovered while implementing that wasn't in the audit]
(or "None")

FOLLOW-ON ITEMS:
- [anything to add to the planning backlog or escalate to the roadmap]
(or "None")

DOCUMENTATION UPDATES NEEDED:
- [any CLAUDE.md, README, or inline docs to update]
(or "None")
---END IMPLEMENTATION SUMMARY BLOCK---

§4

      
Cross-Module Dependency Management

      
Purpose: Prevent implementation changes from silently breaking other modules. Pre-check runs before High/Very High risk changes. Regression check runs after changes, before merging.

    

    
⚠
Treat the Pre-Implementation Check as mandatory for any High or Very High risk change. Discovering breakage after the fact in a 100k+ line codebase costs significantly more than a 5-minute check beforehand.


    
Pre-Implementation Dependency Check

    
📌
Input: Systems Map summary + the specific planned changes from the Implementation Handoff Block.

    

      

        
Pre-Implementation Check

        

          Before High/Very High risk changes
          
          
        

      

      

      
Do not make any changes to any files yet.

[PASTE SYSTEMS MAP SUMMARY HERE, INCLUDING THE INTER-MODULE DEPENDENCY MAP]

I am planning to make the following changes as part of implementing audit findings:
[LIST EACH PLANNED CHANGE WITH THE FILE AND FUNCTION/AREA IT AFFECTS]

Before I make any of these changes:
1. Identify every file outside the current scope that imports from, calls into, or otherwise depends on the specific functions, modules, constants, or data structures being changed
2. For each dependency found, describe the nature of the dependency and what would break or need updating if the change proceeds as described
3. Flag any changes where the impact cannot be determined from the systems map alone and a deeper read of the dependent file is needed
4. Recommend any sequencing — if Change A should happen before Change B to avoid intermediate breakage, say so

Do not begin implementation until this check is complete.

    


    

    
Post-Implementation Regression Check

    
📌
Input: Systems Map summary + a summary of what was actually changed (from the Implementation Summary Block).

    

      

        
Post-Implementation Regression Check

        

          After implementation, before merge
          
          
        

      

      

      
Do not make any changes to any files during this session.

[PASTE SYSTEMS MAP SUMMARY HERE, INCLUDING THE INTER-MODULE DEPENDENCY MAP]

[PASTE IMPLEMENTATION SUMMARY BLOCK FROM THE IMPLEMENTATION SESSION]

Based on the systems map and your knowledge of the codebase:

1. Identify which modules outside the changed scope could have been affected by these changes
2. For each, describe specifically what could have broken and where to verify (file and function/area)
3. For each risk item, explicitly confirm whether the risk materialized or was negated by other factors — cascade configs, zero callers, idempotent operations, existing indexes, default values. Don't just list risks; validate them.
4. Flag anything that warrants a targeted re-audit of the dependent module before the next implementation session
5. Note any documentation (CLAUDE.md, README, roadmap) that should be updated to reflect these changes

Prioritize the verification list by likelihood of breakage. Then produce a FOLLOW-ON AUDIT ITEMS block:

---FOLLOW-ON AUDIT ITEMS---
- [File: function/area] — [what to check, why it was flagged, which change surfaced it]
(repeat for each item, or "None")
These become explicit inputs for the next audit session on the affected modules.
---END FOLLOW-ON AUDIT ITEMS---

Post-Cycle Reflection

          

            Same session as regression check
            
          

        

        
Now that the regression check is complete, provide an honest post-cycle assessment. Do not make any changes to any files.

For each action completed this cycle, answer two questions:

1. "Would this bug have actually fired in production this month?"
   Answer YES (real production bug — currently-reachable code path, realistic load, active user scenario) or NO (speculative, defensive, future-proofing, dead code, or zero-caller path).
   Be specific: if YES, describe the trigger scenario. If NO, say why it wouldn't have fired.

2. "Did this action introduce a new failure mode, documented or not?"
   Answer YES or NO. If YES: describe the new failure mode, whether it is better or worse than what it replaced, and under what conditions it would fire. Do not bury this in a "tradeoffs" section — if the post-cycle state is worse under any realistic scenario, that is a regression and must be counted as one.

Tally:
- Production bug fixes (YES to question 1): [count]
- Speculative/defensive fixes (NO to question 1): [count]
- Actions that introduced new failure modes (YES to question 2): [count]
- Net score: [production fixes] − [new failure modes] = [net]

Honest impact summary — answer each directly:
- What actually changed for a user of this application right now?
- What changed for the next developer working in this subsystem?
- What became safer under scale or concurrent load that wasn't safe before?
- Was any effort spent on dead code, zero-caller paths, or future-proofing that won't be exercised for months?

Invariant growth — answer this:
- "What invariants does this cycle establish that the next Verification Pass should probe?"
  List any rules that must now hold as a result of this cycle's changes. These become candidates for the invariant library. Format each as: [proposed ID] | [one-sentence rule] | [which subsystem/seam it guards] | [how to verify: code read, test, or assertion]

End with:
- One sentence: the single most structurally significant change in this cycle
- One sentence: the finding that should have been deferred — lowest practical impact relative to implementation cost
- Any actions where a design decision produced a tradeoff worth documenting in CLAUDE.md (e.g. "chose graceful degradation over hard failure — see gotcha X")

      

    

  


  
  

    
§4v

      
Independent Verification Pass

      
Purpose: Independent verification of the cycle's work in a fresh session with no memory of the implementation. Probes invariants, counts regressions with a hard definition, checks test coverage quality, and produces a VERIFICATION BLOCK used as input for §6a Health Synthesis.

    

    
⚠
Must run in a completely fresh session. The verifier must have zero context from the implementation session. If you participated in the implementation, start a new session with no prior conversation. This prevents the implementer from grading its own work — the core problem this phase solves.

    
📌
Inputs: Paste the Implementation Summary Block from §3 and the Cycle Summary Block from /reflect. The verifier re-probes invariants via code reads and targeted test runs — not prose assessments.

§5

      
Roadmap & Future Possibilities

      
Purpose: Strategic planning across four time horizons. Run after 2–3 audit sessions or at natural milestones. Tiers 1–3 must be grounded in actual audit findings. Tier 4 is explicitly exploratory.

    

    

      

        
Roadmap Prompt

        

          After 2–3 audit cycles
          
          
        

      

      

      
Do not make any changes to any files during this session.

[PASTE SYSTEMS MAP SUMMARY HERE]
[OPTIONAL: PASTE A SUMMARY OF FINDINGS FROM RECENT AUDIT SESSIONS]

Based on the current state of the codebase and project documentation, produce a roadmap organized into four tiers.

For tiers 1–3, ground every item in a specific finding, gap, or architectural reality from the audit or systems map. Do not include wishlist items that have no basis in what was actually found.

For tier 4, you have explicit permission to be exploratory. Think about the project's domain, purpose, and trajectory — not just its current architecture. Suggest directions that may require significant rethinking but could be genuinely valuable.

---

Tier 1 — Short-term (days to weeks):
Concrete, well-scoped improvements actionable within the current architecture. Include:
- Specific bug fixes and tech debt items from recent audits
- Quick wins with outsized impact on stability or velocity
- Anything blocking other planned work

Tier 2 — Medium-term (weeks to months):
Structural improvements, feature additions, or refactors that require planning but build on what exists. Include:
- Architectural improvements surfaced by the audit
- Feature completions or hardening of existing capabilities
- Integrations or enhancements with clear scope

Tier 3 — Long-term (months+):
Significant capability expansions, architectural evolution, or scaling work. Include:
- Systemic changes that are too large for a sprint but clearly necessary
- Scaling infrastructure if growth is expected
- Major new capability areas grounded in the project's direction

Tier 4 — Future possibilities (exploratory):
Broader directional ideas not constrained by current architecture. Consider:
- How this project could evolve if the core worked perfectly
- Adjacent capabilities that would compound the value of what already exists
- Emerging patterns in the domain (technical or market) worth watching
- Ideas that might seem ambitious but are directionally correct

---

For Tiers 1–3: provide 3–5 items each, with a one-line rationale and effort estimate.
For Tier 4: provide 3 distinct directions (not variations on the same theme), with a short paragraph each explaining the idea and why it's worth considering.

End with: the one strategic bet you'd prioritize if resources were limited, and why.

§6a

      
Project Health Synthesis

      
Purpose: The accurate, benchmarkable holistic score on two axes. Axis A (Vertical) scores subsystem health. Axis B (Horizontal) scores cross-cutting bug-shape posture. Takes three inputs per subsystem: Session Handoff + Cycle Summary + Verification Block. Run only after a full audit cycle — this is your progress benchmark. Never compare Pulse scores to Synthesis scores.

    

    
⚠
Requires three blocks per subsystem: Session Handoff Block (§1), Cycle Summary Block (/reflect), and Verification Block (§4v). Do not run mid-cycle or without these inputs — the score will be unreliable and not comparable to future cycles. The Verification Block is critical for accurate Axis B scoring.

Tips

      
Best Practices

      
Key habits that compound across sessions.

    

    

      
Paste the systems map every session
Claude Code has no memory between sessions. Without it, it re-derives architecture and may contradict prior conclusions. A 3–5 bullet summary is enough.

      
One subsystem group per session
Scope creep mid-audit produces shallow findings. If Claude Code drifts into another module, redirect it explicitly.

      
Confidence levels matter
When Claude Code rates a finding as low confidence, verify manually before acting. At 100k+ lines it will sometimes infer behavior from partial context.

      
Save audit outputs to Archive
Keep a running log of session outputs. When running the Roadmap or Synthesis prompts, paste in findings from all sessions — not just the most recent.

      
Dep. check is mandatory for High risk
The audit surfaces dependencies — but breakage only materializes when you act. Treat §4 Pre-Implementation Check as mandatory, not optional.

      
Never compare Pulse to Synthesis
They're not on the same scale. Pulse is directional and shallow by design. Only compare Synthesis to Synthesis when tracking progress over time.

      
Fresh session for planning (large codebases)
For 110k+ line scopes, auditing a complex module consumes significant context. Start a fresh session for planning with the audit output pasted in.

      
Keep CLAUDE.md lean and current
Recommended sections: Project Overview · Architecture · Key Files & Directories · Conventions & Patterns · Project Health (pointer to PROJECT_HEALTH.md) · Known Issues / Active Work. Don't let it become a notes dump.
